# 一、整体思路先理解

你现在的 `ets` 目录，本质上已经分成了 5 层：

```text
pages        页面入口层
components   可复用界面组件层
models       数据模型层
utils        工具逻辑层
constants    常量配置层
```

你可以把它理解成：

- `pages`：这个页面长什么样，由谁拼起来
- `components`：页面里能复用的小部件
- `models`：数据长什么样
- `utils`：功能逻辑怎么处理
- `constants`：一些固定配置放哪

这样后面你做大了，不会所有代码都堆在 `Index.ets` 里。

------

# 二、pages：页面入口层

## 1. `pages/Index.ets`

这是你的**首页**，也可以理解成“笔记管理主页”。

### 它现在主要负责：

- 左侧边栏整体布局
  - 我的笔记
  - 文件夹
- 右侧内容区整体布局
  - 标题
  - 搜索框
  - 新建笔记入口
  - 笔记列表
- 当前选中了哪个分类 / 哪个文件夹
- 当前右侧应该显示哪些笔记
- 长按文件夹后弹出的菜单
- 点击文件夹后切换右侧内容

### 你可以把它理解为：

**整个首页的总控文件**

它负责“拼装页面”，但理论上不应该承担太多底层业务逻辑。

### 后面还可以继续拆的内容：

以后你完全可以继续把它拆成：

- 左侧栏组件
- 文件夹列表组件
- 右侧笔记区组件
- 文件夹菜单组件

现在之所以还比较大，是因为我们正处于“能跑 + 先把产品骨架搭起来”的阶段。

------

## 2. `pages/EditorPage.ets`

这是你的**编辑器页面**，也就是点进某条笔记后进入的页面。

### 它现在主要负责：

- 接收首页传过来的笔记信息
- 维护编辑器状态
- 切换源码模式 / 渲染模式
- 组合顶部栏、底部栏、源码编辑区、渲染编辑区
- 控制 RichEditor 和 Markdown 文本之间的同步

### 你可以把它理解为：

**编辑器页面总控文件**

它不应该亲自实现所有编辑细节，而是负责把几个编辑器组件组织起来。

### 后面它会继续负责：

- 工具栏点击插入 Markdown 语法
- 保存笔记
- 自动保存
- 编辑器模式切换
- 预览 / 分屏切换

------

# 三、components：可复用组件层

这里放的是“页面里的零件”。

------

## 3. `components/EditorTopBar.ets`

这是**编辑器顶部栏组件**。

### 它一般负责：

- 返回按钮
- 笔记标题显示
- 编辑模式状态显示
- 顶部操作按钮

### 为什么要单独拆出来？

因为顶部栏是一个很明确的 UI 块，单独拆出来以后：

- `EditorPage.ets` 更清爽
- 后面想改顶部栏样式很方便
- 后面加“保存”“分享”“更多菜单”会更容易

### 以后可以扩展：

- 保存按钮
- 导出按钮
- 更多菜单
- 当前是否已保存提示

------

## 4. `components/EditorBottomBar.ets`

这是**编辑器底部栏组件**。

### 它现在/未来一般负责：

- 字数统计
- 当前模式显示
- 编辑器切换按钮
- 预留快捷操作区

### 为什么有它？

底部状态条和工具条通常是一个独立区域，后面功能会越来越多，不适合继续塞在页面里。

### 以后很适合加：

- 当前字数
- 当前行列
- 阅读时间
- Markdown / 预览切换
- 快捷语法工具栏

------

## 5. `components/MarkdownSourceEditor.ets`

这是**Markdown 源码编辑组件**。

### 它负责什么？

简单说，就是：
**纯文本模式下的编辑器**

用户直接看到并编辑 Markdown 原文，例如：

```md
# 标题
**粗体**
- 列表
```

### 它和 `EditorPage.ets` 的关系

- `EditorPage.ets` 负责决定“现在是不是源码模式”
- `MarkdownSourceEditor.ets` 负责真正显示和编辑源码文本

### 以后可以扩展：

- 光标定位
- 选中文本加粗
- Tab 缩进
- 自动补全 Markdown 标记
- 行号显示

------

## 6. `components/MarkdownRenderEditor.ets`

这是**渲染编辑组件**。

### 它负责什么？

简单说：
**富文本/渲染后的编辑视图**

它一般配合 `RichEditor` 使用，让用户看到更接近“排版后的效果”，而不是纯 Markdown 原文。

### 为什么它重要？

因为你目标是做一个像 Typora 一样“好用”的 Markdown 编辑器，最终就需要：

- 源码视图
- 渲染视图
- 甚至所见即所得编辑

这个组件就是这条路线的基础。

### 以后可以扩展：

- 实时渲染标题、引用、列表
- 行内加粗高亮
- 渲染后的点击编辑
- 更高级的富文本同步

# 四、models：数据模型层

这里放的是“数据长什么样”的定义。

------

## 7. `models/AppFolderItem.ets`

这是**文件夹数据模型**。

### 它一般定义：

一个文件夹对象需要有哪些字段，比如：

- `id`
- `name`
- `count`
- `children`
- `expanded`
- `parentId`

### 为什么要单独建？

因为“文件夹”这个东西会在很多地方用到：

- 左侧文件夹列表
- 长按菜单操作
- 新建子文件夹
- 移动文件夹
- 删除文件夹
- 展开/折叠树结构

统一成模型后，整个项目的数据结构会更稳定。

------

## 8. `models/AppMenuItem.ets`

这是**左侧菜单项模型**。

### 它一般表示：

“全部笔记 / 最近笔记 / 我的收藏 / 最近删除” 这类菜单项。

### 一般字段可能有：

- `id`
- `title`
- `icon`
- `count`
- `type`

### 为什么需要它？

因为这些菜单项本质上和“文件夹”不是一类数据。
如果混在一起，会越来越乱。

所以要单独建一个“菜单项”模型。

------

## 9. `models/AppNoteItem.ets`

这是**笔记数据模型**。

### 它非常重要

因为整个应用核心就是“笔记”。

### 一般会包含：

- `id`
- `title`
- `content`
- `updateTime`
- `favorite`
- `folderId`
- `summary`
- `cover`
- `deleted`

### 它的作用：

右侧笔记列表、编辑器页面、收藏状态、笔记筛选，几乎都会依赖它。

### 后面会继续演化：

以后它可能会扩展成：

- 创建时间
- 更新时间
- 标签
- 是否置顶
- 是否加密
- 预览摘要
- Markdown 原文
- HTML 渲染缓存

------

## 10. `models/MarkdownNode.ets`

这是**Markdown 解析节点模型**。

这个文件是你以后“语法能力”的关键基础之一。

### 它的作用是什么？

当你把 Markdown 文本解析后，不是直接乱渲染，而是先变成一棵结构化节点树。

比如：

```md
# 标题
这是 **粗体** 文本
```

会先变成类似这样的结构：

```ts
heading(level=1, text="标题")
paragraph(
  text="这是 ",
  bold("粗体"),
  text=" 文本"
)
```

### 为什么重要？

因为后面你要做：

- 实时预览
- 导出 HTML
- 渲染标题样式
- 渲染列表样式
- 做目录大纲

都需要先有统一的节点结构。

### 这是整个 Markdown 内核的“数据基石”。

# 五、utils：工具逻辑层

这里放的是“处理逻辑”，不是页面，不是数据。

------

## 11. `utils/markdownBlockParser.ets`

这是**Markdown 块级解析器**。

### 它负责什么？

处理 Markdown 的“整块内容”，比如：

- 标题 `#`
- 引用 `>`
- 无序列表 `-`
- 有序列表 `1.`
- 代码块 ```
- 分割线 `---`
- 普通段落

### 为什么叫“块级”？

因为 Markdown 里有两类语法：

## 块级语法

一整行或多行构成的结构，例如：

- 标题
- 引用
- 列表
- 代码块

## 行内语法

一行文字内部的结构，例如：

- 粗体
- 斜体
- 删除线
- 行内代码

`markdownBlockParser.ets` 先负责第一层。

### 后面它会越来越重要

因为你真正开始“完善语法”时，第一步就是增强它。

------

## 12. `utils/markdownCommands.ets`

这是**Markdown 命令工具**。

### 它负责什么？

处理“点击按钮后插入 Markdown 语法”的逻辑。

比如以后你点工具栏：

- 加粗
- 斜体
- 标题
- 列表
- 引用
- 代码块

它就负责把当前文本变成：

- `**选中文本**`
- `# 标题`
- `> 引用`
- `- 列表项`

### 为什么要单独放这里？

因为这个逻辑不属于页面，也不属于解析器。
它属于“编辑命令”。

### 它以后会是工具栏的核心支持文件。

------

## 13. `utils/windowHelper.ets`

这是**窗口辅助工具**。

### 它现在主要负责：

- 隐藏系统栏
- 显示系统栏
- 处理编辑器页面全屏 / 沉浸式体验

### 为什么要单独放？

因为这些是系统窗口相关能力，不应该写死在 `EditorPage.ets` 里。

### 以后还能扩展：

- 状态栏显示/隐藏
- 导航栏控制
- 横竖屏辅助
- 屏幕尺寸判断

------

# 六、constants：常量配置层

------

## 14. `constants/editorConstants.ets`

这是**编辑器常量文件**。

### 它负责什么？

放一些固定不变、会被重复使用的配置，比如：

- 默认字号
- 行高
- 间距
- 圆角
- 默认文本
- 模式枚举值
- 一些统一的 UI 尺寸

### 为什么要有它？

因为如果这些数字直接散落在各个文件里，会出现：

- 到处都是 `12`、`16`、`24`
- 你以后改样式要全局找
- 很容易改漏

放到常量文件里后，统一管理会舒服很多。

# 七、当前工程里“谁调用谁”

你可以简单理解成这条关系链：

```text
Index.ets / EditorPage.ets
    ↓
components
    ↓
models + utils + constants
```

也就是说：

## 页面层

负责“用谁”

## 组件层

负责“怎么显示”

## 模型层

负责“数据长什么样”

## 工具层

负责“逻辑怎么处理”

## 常量层

负责“固定配置放哪里”
